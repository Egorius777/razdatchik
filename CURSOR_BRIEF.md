# Бриф для Cursor: сервис учёта «раздатчик ↔ репетиторы» (Telegram Mini App)

> Этот документ — техническое задание для Cursor. Реализуй проект **с нуля и максимально
> автономно**: принимай разумные решения сам, не останавливайся ради уточнений у пользователя.
> Где значение неизвестно (токены, домен) — используй переменные окружения с заглушками и опиши
> их в README, но не блокируйся. В конце прогони сборку, typecheck и сид-самопроверку.

## 1. Что это и зачем

У репетиторов есть раскрученные аккаунты на агрегаторах (Профи.ру и т.п.). Часть учеников они
передают другим репетиторам. Деньги от клиентов приходят на карту **раздатчика**, а он в конце
недели распределяет доход: оставляет комиссию, остальное переводит репетитору, который реально
вёл уроки.

Сервис должен: репетитор отмечает проведённые уроки и оплаты (с чеками) → раздатчик подтверждает
получение денег → в конце недели раздатчик видит «кто и когда должен был заплатить и пришло ли»
и «кому и сколько перевести за вычетом комиссии».

**Граница (важно):** приложение — это **учётная книга (ledger)**. Оно НЕ двигает деньги, не
подключается к банкам, не проводит платежи. Все переводы люди делают сами вручную; сервис лишь
фиксирует и считает. Никаких платёжных шлюзов.

## 2. Параметры (зафиксировано)

| Параметр | Значение |
|---|---|
| Платформа | **Telegram Mini App** (вход по аккаунту Telegram, без паролей) |
| Доступ | **Закрытый**: репетиторы входят только по приглашению; несколько раздатчиков, каждый со своей изолированной сетью; публичной регистрации нет |
| Комиссия | **Индивидуально под ученика**: процент ИЛИ фиксированная сумма с урока |
| Кто вносит | **Репетитор вносит уроки/оплаты → раздатчик подтверждает** |
| Хостинг + БД | **Railway** (приложение + Postgres + том для файлов) |
| UI | **Magic MCP (21st.dev)** + UI/UX Pro: чисто, современно, mobile-first под Telegram |
| Валюта | Рубли (₽) |
| Период расчёта | Неделя, по умолчанию Пн–Вс (настраивается) |

## 3. Технологический стек (под Railway)

- **Next.js (App Router) + TypeScript + TailwindCSS** — единое приложение: Mini App (фронт) +
  API routes (бэкенд) + вебхук бота + cron-обработчик.
- **Prisma + PostgreSQL** — Railway Postgres (`DATABASE_URL` выдаёт Railway).
- **grammY** — Telegram-бот (вебхук на API route).
- **Telegram WebApp SDK** на фронте; авторизация через валидацию `initData`.
- **Хранение чеков:** Railway **Volume**, смонтированный в `/data` (файлы в
  `/data/receipts/{workspaceId}/...`), отдаются через защищённый API route.
  *Альтернатива (если решишь — ок): Cloudflare R2 / S3-совместимое хранилище.*
- **Cron:** Railway Cron вызывает защищённый эндпоинт недельной сводки.

Один репозиторий, один сервис на Railway + Postgres + Volume. Без Supabase/Vercel.

## 5. Роли и доступ (контроль — на уровне приложения)

Без Supabase RLS — авторизацию проверяй **в каждом API route** на сервере:

- **Раздатчик (distributor)** — владелец workspace. Видит всех репетиторов, всех учеников, все
  уроки и оплаты; подтверждает получение денег; видит недельные выплаты, отмечает «выплачено».
- **Репетитор (tutor)** — приглашённый участник. Видит **только своих** учеников/уроки/оплаты;
  создаёт уроки, отмечает оплату (+чек); видит свою статистику и заработок.
- Раздатчик может и сам вести уроки (назначить ученика на себя) — вторичный сценарий.
- Изоляция между раздатчиками — по `workspaceId`; репетитор ограничен ещё и `tutorId = self`.

## 6. Авторизация (Telegram initData)

1. Фронт получает `window.Telegram.WebApp.initData` → `POST /api/auth/telegram` с этим значением.
2. Сервер валидирует подпись: `secret = HMAC_SHA256("WebAppData", BOT_TOKEN)`, проверяет `hash`
   по отсортированной `data_check_string`; проверяет свежесть `auth_date`.
3. При успехе — upsert `User` по `telegramId`; находит memberships; выдаёт **app-JWT** в httpOnly
   cookie (`userId` внутри).
4. Выбор workspace: 1 членство → авто; несколько → выбор; ноль → онбординг (создать workspace как
   раздатчик ИЛИ ввести код приглашения).
5. Deep-link приглашения: `/start <inviteCode>` в боте и/или ввод кода в Mini App.

## 7. Модель данных (Prisma)

- **Workspace**: `id`, `name`, `ownerUserId`, `cardDetails`, `settlementWeekday` (def 1=Пн),
  `createdAt`.
- **User**: `id`, `telegramId BigInt @unique`, `firstName`, `lastName?`, `username?`, `photoUrl?`,
  `createdAt`.
- **Membership**: `workspaceId`, `userId`, `role` (`Distributor|Tutor`),
  `status` (`Active|Invited|Removed`), `@@unique([workspaceId, userId])`.
- **Invite**: `workspaceId`, `code @unique`, `createdById`, `expiresAt?`, `usedById?`, `usedAt?`.
- **Student**: `workspaceId`, `tutorId`, `name`, `contact?`, `notes?`, `defaultPrice Decimal`,
  `commissionType` (`Percent|Fixed`), `commissionValue Decimal`,
  `status` (`Trial|Active|Paused|Archived`).
- **Lesson**: `workspaceId`, `studentId`, `tutorId`, `scheduledAt`,
  `status` (`Planned|Done|Cancelled|NoShow`), `isTrial Bool`, `price Decimal` (снимок),
  `commissionType`/`commissionValue` (снимок), `commissionAmount Decimal`, `tutorAmount Decimal`,
  `notes?`, `createdById`, `createdAt`, `updatedAt`.
- **Payment** (входящие, клиент → карта раздатчика): `workspaceId`, `studentId`, `tutorId`,
  `amount Decimal`, `status` (`Reported|Confirmed|Disputed`), `receiptFileId?`,
  `reportedById`, `reportedAt`, `confirmedById?`, `confirmedAt?`, `expectedAt?`, `note?`.
- **PaymentLesson**: `paymentId`, `lessonId`, `@@id([paymentId, lessonId])` (одна оплата может
  покрывать несколько уроков; в UI по умолчанию 1 урок = 1 оплата).
- **Payout** (выплата раздатчик → репетитор за период): `workspaceId`, `tutorId`,
  `periodStart`, `periodEnd`, `gross Decimal`, `commissionTotal Decimal`, `netAmount Decimal`,
  `lessonCount Int`, `status` (`Pending|Paid`), `paidAt?`, `paidById?`,
  `@@unique([workspaceId, tutorId, periodStart])`.
- **ReceiptFile**: `workspaceId`, `path` (на томе), `mime`, `size`, `uploadedById`, `createdAt`.

**Снимки (snapshot):** копируй `price`, `commissionType`, `commissionValue` в `Lesson` в момент
создания/подтверждения — чтобы изменение ставки у ученика задним числом не меняло прошлые уроки.

## 8. Расчёт денег

Для урока:
- процент: `commissionAmount = round(price * commissionValue / 100)`;
- фикс: `commissionAmount = min(commissionValue, price)`;
- `tutorAmount = price − commissionAmount`.

Витрины:
- **Входящие (раздатчик):** оплаты по ученику/репетитору со статусом — «от кого и когда должны
  прийти деньги и пришли ли».
- **Выплаты за неделю (раздатчик):** по репетитору `Σ tutorAmount` по подтверждённым урокам
  периода — «кому и сколько перевести».
- **Заработок репетитора:** свои уроки, `Σ tutorAmount`, что ждёт подтверждения.

Контрольный пример: ученик 2000 ₽, комиссия 25%, 3 урока → раздатчику 1500, репетитору 4500.

## 9. Экраны (role-aware, через Magic MCP)

**Репетитор:**
- Главная: эта неделя — мои уроки, заработок, что ждёт подтверждения.
- Ученики: список + карточка (цена, комиссия, статус), добавить/редактировать.
- Урок: добавить/отметить «проведён»; отметить оплату + загрузить чек.
- История/заработок.

**Раздатчик:**
- Дашборд: входящие за неделю (кто/сколько/статус), к выплате по репетиторам, итоги, KPI-карточки.
- Очередь подтверждений: оплаты `Reported` → подтвердить/оспорить.
- Репетиторы: список + пригласить (ссылка/код). Ученики: все.
- Выплаты: по неделям, отметить «выплачено», экспорт CSV.
- Настройки: карта, день недели расчёта.

Общее: нижняя таб-навигация (набор табов зависит от роли), статус-бейджи, состояния
loading/empty/error, тёмная/светлая тема Telegram.

## 10. Бот и уведомления

- Вебхук бота: `POST /api/bot` (grammY). Команды: `/start` → приветствие + кнопка `web_app`
  «Открыть приложение»; `/start <code>` → вступление в workspace по приглашению.
- Скрипт `npm run set-webhook` ставит вебхук на `${PUBLIC_APP_URL}/api/bot`.
- Cron-эндпоинт `POST /api/cron/weekly-settlement` (защита по заголовку `CRON_SECRET`):
  по каждому workspace считает Payout за прошедшую неделю из подтверждённых данных, апсертит
  `Payout`, шлёт раздатчику сводку (входящие + кому платить) и каждому репетитору его заработок.
- Railway Cron: расписание (напр. Пн 09:00) дергает эндпоинт.

## 12. Самопроверка (Cursor выполняет перед сдачей)

1. `prisma migrate` применяется без ошибок; `next build` и `tsc --noEmit` зелёные.
2. Сид-скрипт создаёт: workspace, раздатчика, репетитора, ученика (2000 ₽, 25%), 3 урока.
3. Юнит-тест расчёта: 3 урока → `tutorAmount` суммарно 4500, комиссия 1500.

6. Загрузка/отдача чека через том работает и закрыта авторизацией.

## 13. Допущения (Cursor может опираться, пользователь поправит при необходимости)

- Приложение — только учёт, деньги не двигает.
- Валюта — рубли; период — неделя Пн–Вс (настраивается в workspace).
- Одна оплата может покрывать несколько уроков; в UI по умолчанию 1 урок = 1 оплата.
- Раздатчик может быть и репетитором.
- Доступ закрытый: репетиторы только по приглашению.
- Хранение чеков — Railway Volume (если удобнее — S3/R2, на усмотрение Cursor).
